# Calendar Views, Crop Scheduling & CropDB Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add a horizontal Gantt-style planting timeline, crop scheduling engine, auto-compiled task system with rescheduling, a guided starting flow, and a bundled CropDB dataset to Jninty.

**Architecture:** Extend the existing PouchDB data layer with three new document types (`PlantingSchedule`, `ScheduleTask`, `CustomCrop`). Build a pure scheduling service that computes task dates from CropDB variety data. Render the timeline with a custom DOM renderer + @dnd-kit for drag-to-reschedule. Wire the existing monthly calendar and task list to consume schedule-derived data alongside manual data.

**Tech Stack:** React 18, TypeScript (strict), PouchDB v9, @dnd-kit/core + @dnd-kit/sortable, date-fns, MiniSearch, Zod, Tailwind CSS v4

## 2. New PouchDB Document Schemas

### Design Decision: Separate `scheduleTask` Doc Type vs. Extending Existing `task`

**Recommendation: Use a separate `scheduleTask` doc type.**

Rationale:
1. **Schema divergence** — Schedule tasks have fields that manual tasks never use (`plantingScheduleId`, `taskType`, `scheduledDate`, crop-aware instructions) and manual tasks have fields schedule tasks never use (`recurrence`, `ruleId`, `isAutoGenerated`). A discriminated union inside one schema adds complexity and optionality that Zod `.strict()` makes cumbersome.
2. **Query isolation** — With the single `["docType"]` index, having separate doc types (`task` for manual, `scheduleTask` for schedule-derived) makes queries faster: `docType: "scheduleTask"` returns only schedule tasks without client-side filtering on a discriminator field.
3. **No breaking changes** — The existing `task` schema, `taskRepository`, `TasksPage`, `usePouchQuery`, and `TaskItem` component all remain untouched. Schedule tasks are additive.
4. **Unified display** — The task list page can query both doc types and merge results in the UI layer, which is the same pattern the knowledge browser already uses to merge built-in + custom entries.

The downside is two repositories to manage, but this matches the existing pattern (12 doc types already) and avoids polluting the manual task schema.

### 2.1 `PlantingSchedule` (new doc type: `plantingSchedule`)

Represents a single crop planting with computed timeline. One schedule = one crop/variety placed on one date in one bed.

```typescript
// src/validation/plantingSchedule.schema.ts

import { z } from "zod";
import { baseEntitySchema, isoDate } from "./base.schema.ts";

export const scheduleDirectionSchema = z.enum(["forward", "backward"]);

export const plantingScheduleSchema = baseEntitySchema.extend({
  // Crop reference
  cropId: z.string().min(1),              // CropDB crop ID (static) or custom crop UUID
  varietyId: z.string().min(1),            // CropDB variety ID or custom variety UUID
  cropSource: z.enum(["builtin", "custom"]),

  // Denormalized crop info (snapshot at creation time for offline display)
  cropName: z.string().min(1),            // e.g. "Tomato"
  varietyName: z.string().min(1),          // e.g. "Cherry"

  // Garden placement
  bedId: z.string().uuid().optional(),     // GardenBed reference
  bedName: z.string().min(1).optional(),   // Denormalized for display

  // Scheduling
  anchorDate: isoDate,                     // The date the user placed the crop
  direction: scheduleDirectionSchema,      // "forward" = seed start date, "backward" = harvest target
  seasonId: z.string().uuid().optional(),  // Season reference

  // Succession planting
  successionGroupId: z.string().uuid().optional(), // Shared ID linking succession plantings
  successionIndex: z.number().int().min(0).optional(), // 0-based index within group

  // Computed date cache (denormalized from variety data at creation time)
  seedStartDate: isoDate.optional(),
  bedPrepDate: isoDate.optional(),
  transplantDate: isoDate.optional(),
  cultivateStartDate: isoDate.optional(),
  harvestStartDate: isoDate,
  harvestEndDate: isoDate,

  // Status
  status: z.enum(["active", "completed", "cancelled"]),
  notes: z.string().min(1).optional(),
}).strict();

export type PlantingSchedule = z.infer<typeof plantingScheduleSchema>;
export type ScheduleDirection = z.infer<typeof scheduleDirectionSchema>;
```

**PouchDB `_id` format:** `plantingSchedule:{uuid}`

**Indexes:** Uses existing `["docType"]` index. Queries filter by `docType: "plantingSchedule"` + client-side filtering by `seasonId`, `bedId`, date ranges.

### 2.2 `ScheduleTask` (new doc type: `scheduleTask`)

An auto-generated task derived from a PlantingSchedule. Never manually created.

```typescript
// src/validation/scheduleTask.schema.ts

import { z } from "zod";
import { baseEntitySchema, isoDate, isoTimestamp } from "./base.schema.ts";

export const scheduleTaskTypeSchema = z.enum([
  "seed_start",
  "bed_prep",
  "transplant",
  "cultivate",
  "harvest",
]);

export const scheduleTaskSchema = baseEntitySchema.extend({
  // Parent reference
  plantingScheduleId: z.string().uuid(),

  // Task identity
  taskType: scheduleTaskTypeSchema,
  title: z.string().min(1),                // Auto-generated: e.g. "Start Cherry Tomato seeds indoors"
  description: z.string().min(1).optional(),

  // Denormalized for display (avoid joins)
  cropName: z.string().min(1),
  varietyName: z.string().min(1),
  bedId: z.string().uuid().optional(),
  bedName: z.string().min(1).optional(),

  // Scheduling
  scheduledDate: isoDate,                  // System-calculated due date
  originalDate: isoDate,                   // Original date before any rescheduling

  // Completion
  isCompleted: z.boolean(),
  completedDate: isoDate.optional(),       // Actual completion date (for delta calculation)
  completedAt: isoTimestamp.optional(),     // Timestamp of completion action

  // Ordering within a schedule (for downstream propagation)
  sequenceOrder: z.number().int().min(0),  // 0=seed_start, 1=bed_prep, 2=transplant, 3=cultivate, 4=harvest
}).strict();

export type ScheduleTask = z.infer<typeof scheduleTaskSchema>;
export type ScheduleTaskType = z.infer<typeof scheduleTaskTypeSchema>;
```

**PouchDB `_id` format:** `scheduleTask:{uuid}`

**Sequence order mapping:**

| taskType | sequenceOrder | Color |
|----------|--------------|-------|
| `seed_start` | 0 | Green `#22c55e` |
| `bed_prep` | 1 | Yellow/Gold `#eab308` |
| `transplant` | 2 | Red/Pink `#ef4444` |
| `cultivate` | 3 | Orange `#f97316` |
| `harvest` | 4 | Purple/Magenta `#a855f7` |

### 2.5 Changes to Existing Schemas

**No changes required to the existing `task` schema.** Manual tasks remain exactly as-is. The task list page merges both sources in the view layer.

**No changes to `PlantInstance`, `Planting`, `GardenBed`, `Season`, or `Settings`.** The new scheduling system is additive and does not require modifying existing entities.

### 2.6 Indexing Strategy

The project uses a single `["docType"]` index due to a PouchDB + fake-indexeddb bug (see `src/db/pouchdb/indexes.ts`). This constraint remains.

All new queries use `docType` filtering + client-side post-filtering:
- `docType: "plantingSchedule"` — then filter by `seasonId`, `bedId`, date range in JS
- `docType: "scheduleTask"` — then filter by `plantingScheduleId`, `taskType`, date range in JS
- `docType: "customCrop"` — small dataset, full scan is fine

This is consistent with how all existing repositories work. Performance is acceptable because PouchDB returns all docs of a type and the dataset per user is small (hundreds, not millions).

## 3. Data Flow Architecture

### 3.1 Drop Crop on Calendar → Create Documents

```
User Action: Selects crop/variety from CropPicker, clicks a date on timeline

CropPicker component
  → calls schedulingService.createSchedule(cropId, varietyId, anchorDate, direction, bedId?)

schedulingService.createSchedule():
  1. Look up variety from CropDB (static) or CustomCrop (PouchDB)
  2. Compute all task dates based on direction:
     - If direction = "forward" (seed start date):
       seedStartDate = anchorDate
       bedPrepDate = anchorDate + daysToTransplant - bedPrepLeadDays
       transplantDate = anchorDate + daysToTransplant
       cultivateStartDate = transplantDate + 7 (1 week after transplant)
       harvestStartDate = transplantDate + daysToMaturity
       harvestEndDate = harvestStartDate + harvestWindowDays
     - If direction = "backward" (harvest target):
       harvestEndDate = anchorDate
       harvestStartDate = anchorDate - harvestWindowDays
       transplantDate = harvestStartDate - daysToMaturity
       bedPrepDate = transplantDate - bedPrepLeadDays
       cultivateStartDate = transplantDate + 7
       seedStartDate = transplantDate - daysToTransplant
     - If daysToTransplant is null (direct sow only):
       Skip seedStartDate, bedPrepDate, transplantDate
       directSowDate = anchorDate (forward) or computed backward
       cultivateStartDate = directSowDate + 7
       harvestStartDate = directSowDate + daysToMaturity
       harvestEndDate = harvestStartDate + harvestWindowDays
  3. Create PlantingSchedule document in PouchDB
  4. Create ScheduleTask documents for each applicable task type:
     - seed_start (if variety has daysToTransplant != null and indoorStart = true)
     - bed_prep (if variety has bedPrepLeadDays > 0)
     - transplant (if variety has daysToTransplant != null)
     - cultivate (always)
     - harvest (always)
  5. Return { schedule, tasks } for immediate UI update

PouchDB change feed triggers usePouchQuery re-renders
  → Timeline, Monthly, Yearly views update automatically
```

### 3.2 Mark Task Late → Trigger Rescheduling

```
User Action: Marks a ScheduleTask complete after its scheduledDate

scheduleTaskRepository.complete(taskId):
  1. Set isCompleted = true, completedDate = today, completedAt = now()
  2. Calculate delta = differenceInDays(completedDate, scheduledDate)
  3. If delta > 0 (late completion):
     a. Query all ScheduleTask docs with same plantingScheduleId
     b. Filter to tasks where sequenceOrder > completedTask.sequenceOrder AND isCompleted = false
     c. For each downstream task:
        - scheduledDate = addDays(scheduledDate, delta)
     d. Batch update downstream tasks in PouchDB
     e. Update parent PlantingSchedule's cached dates
  4. If delta <= 0 (on time or early): no propagation
  5. Return { updatedTasks, delta } for toast message

UI layer:
  → usePouchQuery detects changes, re-renders task list + calendar
  → Toast: "Transplant and Harvest dates shifted by {delta} days. Tap to review."
```

### 3.3 Timeline View → Query and Render Events

```
TimelinePage mounts:
  1. useTimelineData(startDate, endDate) hook:
     a. usePouchQuery(() => scheduleTaskRepository.getByDateRange(startDate, endDate))
     b. usePouchQuery(() => taskRepository.getAll()) — manual tasks for dot overlay
     c. Returns { scheduleTasks, manualTasks, loading }

  2. Group scheduleTasks by plantingScheduleId
  3. For each group, compute bar positions:
     - Bar start = task.scheduledDate (as day offset from month start)
     - Bar end = task.scheduledDate (single-day bars) or span (harvest = harvestEndDate)
     - Color = TASK_TYPE_COLORS[task.taskType]
     - Label = `${task.cropName} ${task.varietyName}` (truncated)

  4. Render:
     - For each month in range: one row
     - Day columns within each month
     - Bars positioned absolutely within the row, spanning correct day columns
     - "Today" vertical line indicator

  5. Drag interaction (via @dnd-kit):
     - onDragStart: store task ID + original date
     - onDragOver: show ghost bar at target date
     - onDragEnd: calculate daysDelta, call schedulingService.rescheduleGroup(scheduleId, daysDelta)
```

### 3.4 Succession Planting Flow

```
User Action: Checks "Repeat" in crop picker, sets interval=14 days, count=4

schedulingService.createSuccessionGroup(cropId, varietyId, anchorDate, direction, bedId, interval, count):
  1. Generate a successionGroupId (UUID)
  2. For i = 0 to count-1:
     a. offsetDate = addDays(anchorDate, i * interval)
     b. Create PlantingSchedule with successionGroupId, successionIndex = i
     c. Create all ScheduleTask documents for this schedule
  3. Return array of { schedule, tasks }

UI: Each succession planting appears as a separate group of bars on the timeline,
    visually connected by their matching crop name and sequential positioning.
```

### 4.3 PouchDB Query Pattern

All new repositories follow the exact same pattern as existing ones:

```typescript
// Hook layer (React)
const scheduleTasks = usePouchQuery(
  () => scheduleTaskRepository.getByDateRange(startDate, endDate),
  [startDate, endDate],
);

// Repository layer (PouchDB)
export async function getByDateRange(start: string, end: string): Promise<ScheduleTask[]> {
  await ensureAllIndexes();
  const result = await localDB.find({
    selector: { docType: "scheduleTask" },
  });
  return (result.docs as PouchDoc<ScheduleTask>[])
    .map(stripPouchFields)
    .filter((t) => t.deletedAt == null && t.scheduledDate >= start && t.scheduledDate <= end);
}
```

The `usePouchQuery` hook's PouchDB change listener ensures live updates when tasks are created, rescheduled, or completed.

### 4.4 Calendar Library Integration

See Section 5 for the full recommendation. Summary: **@dnd-kit for drag + custom DOM renderer for the timeline grid**.

The timeline grid is rendered as standard HTML `<div>` elements with CSS Grid layout. @dnd-kit provides the drag primitives (DndContext, useDraggable, useDroppable). No canvas or SVG — DOM rendering ensures text accessibility, CSS styling with Tailwind, and straightforward mobile touch support via @dnd-kit's built-in touch sensor.

## 5. Calendar Library Recommendation

### Options Evaluated

#### A. react-big-calendar

**Pros:**
- Mature library with month/week/day views
- Built-in event rendering

**Cons:**
- No Gantt/horizontal timeline view — it's designed for traditional calendar grids
- Drag-and-drop is for moving events between time slots, not horizontal bar dragging
- Heavy customization needed to get a months-as-rows layout
- No built-in support for spanning bars across days within a row
- Mobile touch support is limited

**Verdict: Not suitable.** The horizontal timeline layout (months as rows, days as columns, spanning bars) is fundamentally different from what react-big-calendar provides. Adapting it would require more effort than building custom.

#### B. @dnd-kit/core + Custom DOM Renderer

**Pros:**
- @dnd-kit is purpose-built for drag-and-drop with first-class touch support
- Touch sensor works on mobile out of the box (PointerSensor handles both mouse and touch)
- Collision detection algorithms (closestCenter, closestCorners) work for snapping bars to day columns
- The timeline grid itself is custom HTML/CSS (CSS Grid), giving full control over the Gantt layout
- Lightweight: @dnd-kit/core is ~10KB gzipped
- Already proven in production for complex drag scenarios (Kanban boards, sortable lists)
- Accessibility: @dnd-kit provides keyboard drag support via KeyboardSensor

**Cons:**
- Requires building the timeline grid layout from scratch (CSS Grid)
- No pre-built calendar primitives — must compute day columns, month rows, bar positioning manually

**Verdict: Best option.** The timeline grid is straightforward CSS Grid layout. @dnd-kit handles the hard part (cross-platform drag with touch, accessibility, collision detection). This combination gives full control over rendering while offloading drag complexity.

#### C. Custom Canvas/SVG Renderer

**Pros:**
- Maximum rendering control
- Can handle thousands of bars efficiently (if needed)

**Cons:**
- Text rendering in Canvas is poor (no CSS, no accessibility, no text selection)
- SVG text requires manual layout and doesn't use Tailwind
- Touch drag must be implemented from scratch (no library help)
- Accessibility is extremely difficult — screen readers can't read Canvas/SVG content meaningfully
- Significantly more code to write and maintain

**Verdict: Not recommended.** The dataset size (a home gardener's plantings — tens to low hundreds of bars) doesn't justify Canvas/SVG performance. DOM rendering with CSS Grid handles this scale easily. The accessibility and maintainability costs of Canvas/SVG are not worth it.

#### D. @tanstack/virtual + @dnd-kit

**Pros:**
- Virtualization for very long timelines (24-month view)
- Combined with @dnd-kit for drag

**Cons:**
- Adds complexity for a marginal performance gain
- The project already uses react-window; mixing virtualizers is messy
- 24 months = 24 rows — not enough to need virtualization

**Verdict: Unnecessary for current scale.** If performance becomes an issue with 24-month views showing 100+ bars, virtualization can be added later. For now, CSS Grid + @dnd-kit is sufficient.

### Recommendation

**Use @dnd-kit/core + @dnd-kit/utilities with a custom CSS Grid timeline renderer.**

Install: `@dnd-kit/core` (~10KB gzip)

Do NOT install `@dnd-kit/sortable` (bars are positioned absolutely, not sorted in a list).

Configuration:
```typescript
import { DndContext, PointerSensor, KeyboardSensor, useSensor, useSensors } from "@dnd-kit/core";

// PointerSensor handles both mouse and touch events
const sensors = useSensors(
  useSensor(PointerSensor, {
    activationConstraint: {
      distance: 8, // 8px minimum drag distance to prevent accidental drags
    },
  }),
  useSensor(KeyboardSensor),
);
```

### 6.3 MiniSearch Indexing

```typescript
// src/services/cropDBSearch.ts

import MiniSearch from "minisearch";
import { loadCropDB } from "../data/cropdb/index.ts";
import type { CropRecord, CropVariety } from "../data/cropdb/cropdb.types.ts";
import type { CustomCrop } from "../validation/customCrop.schema.ts";

interface CropSearchDoc {
  id: string;         // varietyId
  cropId: string;
  cropName: string;
  varietyName: string;
  category: string;
  source: "builtin" | "custom";
}

let searchIndex: MiniSearch<CropSearchDoc> | null = null;

export function buildCropSearchIndex(customCrops: CustomCrop[]): MiniSearch<CropSearchDoc> {
  const index = new MiniSearch<CropSearchDoc>({
    fields: ["cropName", "varietyName", "category"],
    storeFields: ["cropId", "cropName", "varietyName", "category", "source"],
    searchOptions: {
      boost: { cropName: 2, varietyName: 1.5 },
      fuzzy: 0.2,
      prefix: true,
    },
  });

  // Index built-in crops
  const builtIn = loadCropDB();
  const docs: CropSearchDoc[] = [];

  for (const crop of builtIn) {
    for (const variety of crop.varieties) {
      docs.push({
        id: variety.id,
        cropId: crop.id,
        cropName: crop.commonName,
        varietyName: variety.name,
        category: crop.category,
        source: "builtin",
      });
    }
  }

  // Index custom crops
  for (const crop of customCrops) {
    for (const variety of crop.varieties) {
      docs.push({
        id: variety.id,
        cropId: crop.id,
        cropName: crop.commonName,
        varietyName: variety.name,
        category: crop.category,
        source: "custom",
      });
    }
  }

  index.addAll(docs);
  searchIndex = index;
  return index;
}

export function searchCrops(query: string): CropSearchDoc[] {
  if (!searchIndex) return [];
  return searchIndex.search(query) as unknown as CropSearchDoc[];
}
```

**Initialization:** Called once in `App.tsx` useEffect, after `rebuildIndex()` (the existing MiniSearch for plants/journal). This is a separate MiniSearch instance — the existing search index is unrelated and should not be modified.

### 6.4 Custom Crop Merging

```typescript
// src/hooks/useCropDB.ts

export function useCropDB() {
  const customCrops = usePouchQuery(() => customCropRepository.getAll());

  const allCrops = useMemo(() => {
    const builtIn = loadCropDB();
    if (!customCrops) return builtIn;

    // Custom crops are appended after built-in, tagged with source
    // The CropPicker UI shows them in a "My Custom Crops" category
    return builtIn;
  }, [customCrops]);

  const search = useCallback((query: string) => {
    return searchCrops(query);
  }, []);

  return { allCrops, customCrops, search, loading: customCrops === undefined };
}
```

The `CropPicker` component merges built-in + custom crops in the UI. Built-in crops are read from the in-memory cache. Custom crops are queried from PouchDB and displayed in a separate "My Crops" section or mixed into search results with a "Custom" badge.

- Modify: `src/db/pouchdb/repositories/scheduleTaskRepository.ts` — enhance `complete()` method

When `scheduleTaskRepository.complete(id)` is called:
1. Set `completedDate = today`, `isCompleted = true`, `completedAt = now()`
2. Check if `completedDate > scheduledDate` (late)
3. If late: call `schedulingService.propagateLateCompletion(task)`
4. `propagateLateCompletion`:
   a. Calculate `delta = differenceInDays(completedDate, scheduledDate)`

   e. Update parent PlantingSchedule cached dates

## 8. Open Questions & Risks

### Open Questions

| # | Question | Impact | Suggested Default |
|---|----------|--------|-------------------|
| 1 | Should PlantingSchedule link to an existing PlantInstance, or is it independent? The spec says "crop picker" not "plant picker", implying schedule operates on CropDB varieties, not existing PlantInstance records. | Schema design | Keep independent. PlantingSchedule references CropDB, not PlantInstance. Users can optionally link a PlantInstance later. |
| 2 | What happens if a user modifies a CustomCrop variety after creating PlantingSchedules that reference it? | Data integrity | Denormalized fields in PlantingSchedule (cropName, varietyName) are snapshots. Changes to the crop don't retroactively update existing schedules. Display a "source data changed" warning if detected. |
| 3 | How should the CropDB data be populated? 3,000+ varieties is a large dataset to hand-author. | Content | Start with ~60 high-quality varieties (Task 1.2). Expand using community contributions, AI-assisted data generation from public seed catalog data, or a data import tool. |
| 4 | Should the timeline view show manual tasks as well, or only schedule-derived tasks? | UI design | Show schedule tasks as bars. Show manual tasks as small diamond markers (matching existing calendar dot style). Keep them visually distinct. |
| 5 | What's the cultivate task — is it a single date or a date range? | Schema | Model as a single start date. Cultivating is ongoing; the task represents "begin regular cultivation" not a specific end date. |
| 6 | Should the starting flow wizard replace the CropPicker quick-drop on the timeline, or coexist? | UX | Coexist. Quick-drop is for experienced users who know what they want. Wizard is for first-time users or complex setups. |

### Technical Risks

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| 1 | **CropDB bundle size** — 3,000 varieties at ~200 bytes each = ~600KB raw JSON. Gzipped ~60-80KB. May impact initial load on slow connections. | Medium | Vite code-splits JSON imports. Service worker caches the bundle after first load. Consider lazy-loading CropDB on first calendar access rather than app startup. |
| 2 | **PouchDB single-index limitation** — All queries filter by `docType` then post-filter in JS. With schedule tasks (potentially hundreds per user per season), full-scan queries may become slow. | Low | Profile with 500+ schedule tasks. If slow, add client-side caching in the repository layer (like CropDB's memory cache). The dataset size for a home gardener is unlikely to hit performance limits. |
| 3 | **@dnd-kit touch drag vs scroll conflict** — On mobile, horizontal scrolling within a month row may conflict with touch drag on bars. | High | Use @dnd-kit's `activationConstraint: { distance: 8 }` to require 8px of movement before drag activates. This allows scroll gestures to work normally and only activates drag on deliberate bar grabs. Test extensively on iOS Safari and Chrome Android. |
| 4 | **CSS Grid timeline rendering on Safari** — Safari has known quirks with CSS Grid sticky positioning and overflow scroll. | Medium | Test on Safari early. Fallback: use `position: sticky` polyfill or restructure layout to avoid sticky + overflow combination. |
| 5 | **Batch PouchDB writes** — Creating a PlantingSchedule + 5 ScheduleTask documents = 6 PouchDB writes. With succession planting (4 plantings = 24 writes), this may be slow. | Low | PouchDB's `bulkDocs` API handles batch writes efficiently. Use `localDB.bulkDocs()` instead of individual `localDB.put()` calls in `createBatch()`. |
| 6 | **Auto-rescheduling cascades** — If a user completes tasks late repeatedly, downstream tasks shift further and further. This could create confusing date jumps. | Medium | Show a clear toast with the delta. Add a "Reset to original dates" action on the PlantingSchedule (restoring all tasks to their `originalDate` values). Limit propagation to only the immediate next uncompleted task, not all downstream (debatable — could also propagate to all). |
| 7 | **Existing calendar refactoring** — `PlantingCalendarPage.tsx` is a 537-line monolith. Enhancing it requires extracting the monthly grid into a reusable component. | Medium | Phase 7 Task 7.1 should start by extracting the monthly grid into `src/components/calendar/MonthlyView.tsx` before adding schedule task data. |
